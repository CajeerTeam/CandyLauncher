# Политика безопасности

Этот документ описывает порядок сообщения об уязвимостях безопасности в **CandyLauncher** — open-source проекте, опубликованном **Cajeer** и разрабатываемом **Candy Interactive**.

## Поддерживаемые версии

Поддержка безопасности предоставляется для версий, явно отмеченных сопровождающими как поддерживаемые.

Используйте таблицу ниже для конкретного репозитория:

| Версия | Поддерживается |
|---|---:|
| Последний стабильный релиз | Да |
| Предыдущий стабильный релиз | По возможности |
| Ветка разработки | По возможности |
| Версии с завершенной поддержкой | Нет |

Сопровождающие могут обновлять эту таблицу для конкретного проекта или release-линейки.

## Сообщение об уязвимости

Не сообщайте об уязвимостях безопасности через публичные issues, discussions, pull requests или социальные сети.

Используйте приватный канал:

```text
security@cajeer.ru
```

Если в репозитории доступен механизм private security advisory, используйте его.

В отчете желательно указать:

- затронутый проект и версию;
- затронутый компонент или модуль;
- класс уязвимости, если он известен;
- предварительные условия атаки;
- шаги воспроизведения;
- proof of concept, если его безопасно предоставить;
- ожидаемое влияние;
- логи, скриншоты или трассировки, если они полезны;
- является ли информация уже публичной;
- предпочтительный способ связи с вами.

Не включайте настоящие production-секреты, персональные данные, приватные ключи, access tokens или пользовательские данные.

## Область рассмотрения

К вопросам безопасности могут относиться, в частности:

- обход аутентификации;
- обход авторизации;
- повышение привилегий;
- удаленное выполнение кода;
- command injection;
- SQL injection;
- cross-site scripting;
- cross-site request forgery;
- server-side request forgery;
- path traversal;
- произвольное чтение или запись файлов;
- небезопасная десериализация;
- утечка секретов;
- небезопасная криптография;
- dependency confusion;
- компрометация supply chain;
- небезопасная загрузка плагинов или расширений;
- escape из sandbox;
- некорректные проверки прав;
- нарушение audit logging для действий, связанных с безопасностью.

## Обычно вне области рассмотрения

Следующие случаи обычно не рассматриваются как уязвимости без демонстрации конкретного влияния на безопасность:

- отчеты автоматических сканеров без анализа;
- отсутствие отдельных security headers без эксплуатируемого сценария;
- self-XSS без влияния на границы привилегий;
- уязвимости, требующие полного административного доступа;
- denial-of-service, требующий нереалистичных ресурсов;
- социальная инженерия;
- физические атаки;
- проблемы в неподдерживаемых версиях;
- уязвимости в сторонних сервисах, не контролируемых проектом;
- публично известные уязвимости зависимостей, которые не эксплуатируются в конфигурации проекта.

## Координированное раскрытие

Cajeer придерживается координированного раскрытия уязвимостей.

Ожидаемый процесс:

1. Исследователь отправляет приватный отчет об уязвимости.
2. Сопровождающие подтверждают получение отчета.
3. Сопровождающие проверяют проблему.
4. Сопровождающие разрабатывают и ревьюят исправление.
5. Сопровождающие выпускают исправленные версии, если это применимо.
6. Сопровождающие публикуют advisory или запись в changelog, если это уместно.

Не раскрывайте уязвимость публично до того, как сопровождающие получили разумную возможность исследовать проблему и выпустить исправление.

## Safe harbor

Cajeer поддерживает добросовестные исследования безопасности.

Добросовестное исследование означает, что вы:

- избегаете нарушения приватности;
- избегаете уничтожения данных;
- избегаете persistence;
- избегаете lateral movement;
- избегаете нарушения доступности сервиса;
- не извлекаете данные сверх необходимого минимума для подтверждения влияния;
- сообщаете о проблеме приватно и своевременно;
- предоставляете сопровождающим разумную возможность исправить проблему.

Эта политика не разрешает тестирование систем, сервисов, аккаунтов, инфраструктуры или данных, которыми вы не владеете и на тестирование которых у вас нет разрешения.

## Обработка секретов

Если вы случайно обнаружили секрет:

1. Остановите тестирование.
2. Не используйте секрет.
3. Не копируйте дополнительные данные.
4. Сообщите об обнаружении приватно.
5. Предоставьте только минимальное подтверждение.
6. Удалите локальные копии после подтверждения получения отчета, если закон не требует иного.

## Обновления безопасности

Исправления безопасности могут выпускаться как:

- patch-релизы;
- hotfix-релизы;
- backport-исправления;
- рекомендации по конфигурации;
- временные mitigations;
- security advisories.

Пользователи должны своевременно обновлять развертывания и изучать release notes, связанные с безопасностью.
