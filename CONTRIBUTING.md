# Участие в разработке CandyLauncher

Спасибо за интерес к участию в разработке **CandyLauncher** — open-source проекта, опубликованного **Cajeer** и разрабатываемого **Candy Interactive**.

Этот документ определяет правила участия в разработке: код, документация, issues, pull requests, изменения, связанные с безопасностью, и общие правила сопровождения проекта.

## Лицензия

Если явно не указано иное, все вклады в этот репозиторий передаются на условиях той же лицензии, что и сам проект:

```text
Apache License 2.0
```

Отправляя вклад в проект, вы подтверждаете, что:

1. У вас есть законное право передать этот вклад.
2. Ваш вклад может распространяться под лицензией проекта.
3. Ваш вклад не содержит заведомо проприетарных, конфиденциальных или незаконно скопированных материалов.
4. Вы понимаете, что принятые изменения могут быть изменены, переработаны, задокументированы или удалены сопровождающими проекта.

## Допустимые виды вклада

Проект может принимать:

- исправления ошибок;
- усиление безопасности;
- улучшения документации;
- улучшения тестового покрытия;
- оптимизацию производительности;
- улучшения доступности;
- локализацию и переводы;
- рефакторинг, повышающий сопровождаемость;
- новые функции, соответствующие дорожной карте проекта.

Изменения могут быть отклонены, если они:

- ломают обратную совместимость без понятного пути миграции;
- добавляют лишние зависимости;
- ухудшают безопасность, наблюдаемость или сопровождаемость;
- создают необоснованную привязку к поставщику;
- противоречат архитектуре проекта;
- дублируют существующую функциональность;
- слишком велики для нормального ревью;
- содержат сгенерированный, обфусцированный или скопированный код без понятного происхождения.

## Перед открытием pull request

Перед отправкой pull request:

1. Проверьте существующие issues и pull requests.
2. Для крупных функций или breaking changes сначала откройте issue.
3. Делайте изменение сфокусированным.
4. Обновляйте документацию при изменении поведения.
5. Добавляйте или обновляйте тесты, если это применимо.
6. Запускайте проверки проекта локально.
7. Убедитесь, что в коммитах нет секретов, токенов, приватных ключей, дампов или учетных данных.

## Именование веток

Рекомендуемый формат имени ветки:

```text
тип/краткое-описание
```

Примеры:

```text
fix/login-rate-limit
feature/plugin-installer
wiki/configuration-guide
security/session-cookie-flags
refactor/storage-adapter
```

Рекомендуемые типы:

- `fix`
- `feature`
- `docs`
- `security`
- `refactor`
- `test`
- `build`
- `ci`
- `chore`

## Стиль коммитов

Используйте понятные и прямые сообщения коммитов.

Рекомендуемый формат:

```text
тип(область): краткое описание
```

Примеры:

```text
fix(auth): validate expired sessions
feature(api): add project token rotation
docs(readme): document installation flow
security(headers): enable strict transport security
```

Коммиты должны быть логически разделены. Не смешивайте несвязанные форматирование, рефакторинг, обновление зависимостей и функциональные изменения в одном коммите.

## Требования к pull request

Pull request должен содержать:

- краткое описание изменения;
- мотивацию изменения;
- сведения о проверке;
- миграционные заметки, если изменилось поведение или конфигурация;
- скриншоты или записи экрана для изменений интерфейса;
- заметки по безопасности для изменений в аутентификации, авторизации, криптографии, сетевом взаимодействии или хранении данных.

Рекомендуемая структура pull request:

```markdown
## Краткое описание

## Мотивация

## Изменения

## Проверка

## Миграционные заметки

## Влияние на безопасность

## Скриншоты
```

## Требования к качеству кода

Изменения должны соответствовать следующим правилам:

- Предпочитайте простой и явный код.
- Избегайте скрытых побочных эффектов.
- Валидируйте входные данные на границах доверия.
- По возможности сохраняйте стабильность публичных API.
- Делайте сообщения об ошибках полезными, но не раскрывайте секреты.
- Предпочитайте структурированное логирование произвольному выводу.
- Сохраняйте совместимость с поддерживаемыми платформами.
- Не добавляйте телеметрию, если она не задокументирована и не настраивается.
- Не добавляйте сетевые обращения, которые не нужны для работы функции.

## Изменения, связанные с безопасностью

К изменениям, связанным с безопасностью, относятся изменения в следующих областях:

- аутентификация;
- авторизация;
- сессии;
- обработка паролей;
- обработка токенов;
- криптография;
- хранение секретов;
- загрузка файлов;
- защита от path traversal;
- удаленное выполнение команд;
- загрузка зависимостей;
- выполнение плагинов;
- сетевой доступ;
- границы sandbox;
- модель прав доступа;
- аудит и логирование действий безопасности.

Для таких изменений:

1. Опишите влияние на безопасность в pull request.
2. Укажите возможные сценарии злоупотребления, если они релевантны.
3. Не ослабляйте безопасные настройки по умолчанию без явного описания.
4. Не раскрывайте нерешенные уязвимости в публичных issues или pull requests.
5. Следуйте процедуре из `SECURITY.md`.

## Зависимости

Новые зависимости должны быть обоснованы.

Перед добавлением зависимости проверьте:

- активно ли она сопровождается;
- совместима ли ее лицензия с лицензией проекта;
- действительно ли она необходима;
- можно ли решить задачу существующим кодом проекта;
- увеличивает ли она поверхность атаки;
- влияет ли она на время запуска, размер пакета или сложность развертывания.

Изменения зависимостей должны обновлять lock-файлы и уведомления о сторонних компонентах, если это применимо.

## Документация

Документация должна быть ясной, учитывать версии и позволять воспроизвести результат.

При добавлении или изменении поведения обновляйте связанные материалы:

- README;
- руководство по установке;
- справочник конфигурации;
- документацию API;
- руководство по миграции;
- заметки по безопасности;
- changelog или release notes, если применимо.

## Сообщения об ошибках и issues

При создании issue укажите:

- версию проекта или commit hash;
- среду выполнения;
- операционную систему и версию;
- шаги для воспроизведения;
- ожидаемое поведение;
- фактическое поведение;
- логи, stack traces или скриншоты, если они полезны;
- релевантную конфигурацию с удаленными секретами.

Не публикуйте секреты, токены, приватные ключи, пароли, production-дампы или приватные пользовательские данные.

## Права сопровождающих

Сопровождающие проекта могут:

- редактировать заголовки, labels, milestones и описания;
- закрывать issues, которые являются дубликатами, устарели, некорректны или выходят за рамки проекта;
- запрашивать изменения перед merge;
- отклонять изменения, не соответствующие направлению проекта;
- выполнять squash, rebase или реструктуризацию коммитов перед merge;
- удалять небезопасные, несопровождаемые или юридически проблемные изменения.

## Стандарты сообщества

Все участники должны соблюдать `CODE_OF_CONDUCT.md`.

Проекты Cajeer должны оставаться техническими, конструктивными и профессиональными.
